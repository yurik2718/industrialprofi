# Фабрика контента — архитектура и канон создания материалов

Единый источник правды о том, как IndustrialProfi производит **глубокие,
структурированные, надёжные материалы для многих профессий** — быстро с ИИ, затем
проверенные экспертом, затем улучшаемые читателями — без потери доверия и **без
того, чтобы основатель был бутылочным горлышком**.

Это и **архитектура** (как устроено и почему так), и **канон** (как правильно
создать качественную профессию по шагам). Промпты-инструменты (`AUTHOR_PROFESSION.md`,
`DEEPEN_LESSON.md`, `LESSON_IMAGES.md`, `QA_REVIEW.md`) — это «ИИ» фабрики; здесь —
карта, которая их связывает.

> **Документ на русском намеренно** (общее правило проекта — доки на английском).
> Всё, что касается создания контента, работает над русскоязычными материалами,
> поэтому канон создания тоже русский. См. `CLAUDE.md → Docs`.

---

## 1. Главный инвариант (его не нарушаем)

> **ИИ ускоряет, эксперты проверяют и публикуют, неизменяемый конвейер защищает.
> ИИ никогда не публикует. ИИ никогда не перетирает то, что тронул человек. Весь
> ИИ работает на этапе авторинга/обслуживания (консоль Claude Code или любой
> внешний LLM) — само приложение остаётся БЕЗ LLM** (нет платной зависимости в
> рантайме, нет стоимости на пользователя), ровно как осознанное решение «без
> чат-бота».

Настоящее узкое место качества-в-масштабе — **ёмкость экспертной проверки**, а не
генерация. ИИ делает черновики дешёвыми; доверие масштабируется людьми. Поэтому
черновики могут копиться в `draft`, но публикация идёт со скоростью, с которой
эксперты успевают проверять. Опубликовать непроверенный ИИ-контент = разрушить
доверие, которое и есть весь ров.

---

## 2. Иерархия контента

Четыре уровня, точный паритет с The Odin Project:

```
Профессия      →  Курс           →  Раздел (stage)        →  Урок
Path              Course             Lesson#stage             Lesson
(Электрик)        (Электромонтаж)    («Правила устройства»)   (ПУЭ гл. 1.7)
/paths/:slug      /courses/:slug     строка-заголовок,        /lessons/:slug
                                     не модель
```

- `Path` (профессия) и `Course` (курс) — настоящие навигируемые модели.
- **Раздел** — это не модель, а строка `Lesson#stage`, по которой уроки
  группируются на странице курса.
- `Lesson` — урок с плоским URL `/lessons/:slug`. `slug` **глобально уникален**
  (это держит индекс БД — см. §5).

Инварианты модели (их не «чинить»): `lessons.path_id` денормализован (= `course.path`,
синхронизируется `before_validation`); `lesson.position` глобальна в пределах
профессии (непрерывные «назад/вперёд» через границы курсов); destroy-цепочка —
`Path → courses → lessons` (курс владеет уроками). Подробно — `CLAUDE.md → Content
architecture`.

## 3. Конвейер (по профессии)

```
   ┌─ авторинг (offline, ИИ) ─────────────┐   ┌─ приложение (без LLM) ──────────┐
   │                                      │   │                                 │
   │  1. Скелет + уроки  AUTHOR_PROFESSION │   │  4. Экспертное ревью в админке  │
   │       │  (формат сидов .md)          │   │       проверка точности/        │
   │       ▼                              │   │       безопасности → ПУБЛИКАЦИЯ  │
   │  2. Глубина по уроку  DEEPEN_LESSON  │   │       (первая правка = freeze)   │
   │       │                              │   │       │                         │
   │       ▼                              │   │       ▼                         │
   │  3. Визуалы  LESSON_IMAGES           │   │  5. Правки читателей →          │
   │       │                              │   │       ревью редактора →         │
   │       ▼                              │   │       ревизия (Wikipedia-style) │
   │   импорт в БД (draft, origin=ai/seed)│   │                                 │
   └──────────────┬───────────────────────┘   └─────────────────────────────────┘
                  │                                          ▲
                  └────────►  БД (источник истины)  ◄────────┘
                       6. QA на масштабе (механика + суждение) — постоянно
```

1. **Скелет + уроки (ИИ, offline).** `AUTHOR_PROFESSION.md` проектирует структуру
   и пишет уроки в формате сидов. Останавливается согласовать roadmap до написания.
2. **Глубина, по одному уроку (ИИ, offline).** `DEEPEN_LESSON.md` — точечная
   доработка; именно поурочный фокус даёт реальную глубину (один прогон на всю
   профессию = поверхностно).
3. **Схемы/инфографика — только где помогают** (`LESSON_IMAGES.md`). **Техсхемы
   обязан проверить эксперт** — тонко неверная схема опасна.
4. **Экспертное ревью (гейт качества).** Редакторы проверяют черновики в админ-
   редакторе и публикуют. Первая правка человека забирает владение
   (`origin: human`) и замораживает урок от будущего ИИ-реимпорта.
5. **Правки читателей (сообщество).** Читатель предлагает правку → ревью редактора
   → ревизия. Self-developing, но модерируемо.
6. **QA на масштабе (постоянно).** Две половины — см. §7.

## 4. Два входа контента в БД

```
  разработчик (репо)                    эксперт (без репо)
  db/seeds/curriculum/<slug>/.md        вставка одного YAML
        │  bin/rails content:import           │  /admin/imports/new
        │  CurriculumImporter                 │  предпросмотр → подтвердить
        │                                     │  CurriculumDocument
        └──────────────┬──────────────────────┘
                       ▼
            ImportUpsert (общее create-or-refresh, §5)
                       ▼
            БД: draft, origin = seed | ai
```

Оба входа делят **один движок решения** (`lib/import_upsert.rb`) и потому ведут
себя одинаково в опасной части. Различаются только тем, что им свойственно
по-настоящему: разработчик задаёт порядок авторитетным деревом файлов; паста
форсирует `draft`, проставляет автора и идёт атомарной транзакцией с предпросмотром.

**Предпросмотр пасты = тот же код, что и импорт, в откатываемой транзакции** — поэтому
превью не может разойтись с тем, что сделает подтверждение.

## 5. Как контент попадает в БД безопасно

### create-or-refresh + заморозка

БД — источник истины. Импортёр — это **подающий конвейер create-or-refresh**,
управляемый `Importable#frozen_for_import?`:

```
строка по slug …
  │
  ├─ новая ───────────────────────────────► СОЗДАЁТСЯ (origin=ai/seed, draft)
  │
  ├─ нетронутая (importer-owned, digest сходится) ─► ОБНОВЛЯЕТСЯ на месте
  │        (так ИИ углубляет свои же черновики при реимпорте)
  │
  └─ замороженная ───────────────────────► ПРОПУСКАЕТСЯ (труд человека не трогаем)
           что замораживает:
             • origin == "human"  (человек создал/забрал владение), ИЛИ
             • есть ревизия       (одобренная правка/админ-правка), ИЛИ
             • digest разошёлся   (тронуты поля, которые ведёт импортёр)
```

Заморозка **по строке, не по поддереву**: замороженная (например, опубликованная)
профессия всё равно обходится, поэтому новые уроки под ней импортируются.

Почему три триггера, а не один: правки человека уходят в `rich_*` (см. ниже) в
обход digest по markdown-колонкам, поэтому одного digest мало — наличие ревизии и
`origin=human` закрывают это независимо. Это намеренный belt-and-suspenders.

### Уникальность slug и защита от потери

`slug` урока/курса/пути **глобально уникален** (плоские URL `/lessons/:slug`).
Отсюда три класса потери данных — все закрыты:

```
коллизия slug с ДРУГОЙ профессией   → CrossPathConflict (отказ, а не тихий перенос)
дубль slug ВНУТРИ одного импорта     → DuplicateSlugConflict (отказ, а не слияние)
вставка дубля одновременно           → ловит UNIQUE-индекс БД
```

Без этого generic-заголовок («Введение») сводился бы к занятому slug, импортёр
нашёл бы чужую строку и **молча перенёс** бы её в импортируемую профессию (или
слил два урока в один). Теперь — громкий отказ с именем slug.

Сид-импорт идёт **атомарно по профессии** (транзакция вокруг каждой): конфликт
откатывает профессию целиком, а не оставляет полпрофессии.

### Чем держится (три слоя — ни один не зависит от дисциплины)

```
БД        уникальные индексы slug · origin NOT NULL default "human" (fail-safe:
          неизвестное происхождение = заморожено) · счётчики NOT NULL · FK
движок    ImportUpsert: заморозка + cross-path + дедуп — в одном месте, оба
          импортёра зовут его
тесты     инварианты зафиксированы (заморозка, отказ от коллизий, fail-safe origin)
          — рефакторинг не сломает их тихо
```

### Дуальное хранение (markdown ↔ rich)

```
markdown-колонки  body / description / task   ← пишет ИМПОРТЁР (ИИ/seed)
rich-текст        rich_body / rich_*          ← пишет ЧЕЛОВЕК (админ-редактор Lexxy)

во вьюхе ПОБЕЖДАЕТ rich, если есть; иначе — markdown (как fallback)
наличие rich = заморозка (импортёр такую строку не трогает)
```

Правило в одну строку: **rich побеждает в показе; наличие rich = заморозка.** Вся
тонкость редактирования растёт отсюда — держи это в голове.

> **Почему НЕТ флага «перетереть правки» (`--force`).** Импортёр пишет markdown, а
> человек правит `rich_*`, который во вьюхе побеждает — наивный `--force` был бы и
> неэффективен (rich всё равно показывается), и разрушителен (сбрасывает
> атрибуцию и ревизию). Инвариант «ИИ не перетирает человека» ценен тем, что
> **безусловен**. Нужно переработать черновик с нуля — удали Path и пересей
> (только для `draft` без данных пользователей); опубликованное правится поурочно
> в админке. Это решение записано — не предлагать заново.

## 6. Канон: как создать качественную профессию по шагам

Полные правила формата и качества — в `AUTHOR_PROFESSION.md` (не дублируем здесь).
Это — карта шагов.

```
1 roadmap → 2 уроки → 3 глубина → 4 визуалы → 5 импорт → 6 проверка → 7 публикация → 8 поддержка
   AUTHOR     AUTHOR    DEEPEN      IMAGES      rake       check+QA     осознанно      reader edits
```

1. **Спроектировать roadmap.** Новый чат → `AUTHOR_PROFESSION.md`. Модель задаёт
   вопросы, предлагает структуру (разделы → уроки, что обязательно/опционально) и
   **останавливается на согласование**. Проверь карту до написания.
2. **Написать уроки** в формат сидов (`db/seeds/curriculum/<slug>/`), `status: draft`:
   ```
   <slug>/
     path.yml                  # title, description, position, status: draft
     01-<раздел>/
       section.yml             # title → станет stage урока
       <урок>.md               # frontmatter → ЗАЧЕМ → тело → ## Задание
   ```
3. **Углубить по одному уроку** (`DEEPEN_LESSON.md`) — там, где тема важная: реальная
   глубина рождается поурочно, а не одним прогоном.
4. **Визуалы — только где помогают** (`LESSON_IMAGES.md`). Брифы лежат в `alt`
   плейсхолдеров. Техсхемы потом проверяет эксперт.
5. **Импортировать** (черновик, отчёт `+N new / K frozen`):
   ```bash
   bin/rails "content:import[<slug>]"   # одна профессия; без slug — всё дерево
   ```
   Реимпорт **обновляет нетронутые ИИ-черновики на месте**, замороженные пропускает.
6. **Проверить:**
   ```bash
   bin/rails test            # машинерия — зелено перед любой правкой кода
   bin/rails content:check   # QA контента: audit + links
   ```
   Плюс **глазами** в админке как черновик и **судящее ревью** (`QA_REVIEW.md`).
7. **Опубликовать осознанно** — это делает человек, не ИИ. Через сиды: `status:
   draft → published` в yml + реимпорт (статус-строка обновится, если не заморожена).
   Через админку: кнопкой на `/admin/paths` / `/admin/courses` (только админ).
8. **Поддержка.** Правки читателей и редакторов копятся как ревизии; реимпорт
   углубляет только нетронутые черновики, человеческое не трогает.

### Формат урока (канон, кратко)

Каждый урок: **ЗАЧЕМ** (`description`, ≤155 символов, в поисковой формулировке —
двойная работа: мотивация + SEO-сниппет) → **ясное оригинальное объяснение**
(`body`) → **типо-уместные ссылки** для углубления. Закрывай урок **по пользе, а
не по шаблону**: теория — блоком самопроверки `> [!ПРОВЕРЬ]`; практика
(`kind: practice`) — бриф-форматом (Цель → Понадобится → `[!ОПАСНО]` → Шаги → Что
сдать → Самопроверка) и только где навык реально требует рук. **Польза важнее
галочки — никогда не добавляй секцию ради секции.** Полная рубрика — в
`AUTHOR_PROFESSION.md`.

## 7. Редактирование — три пути

| Кто | Как | Что происходит |
|-----|-----|----------------|
| **Разработчик** | правит `.md`/`.yml` → `content:import[<slug>]` | обновляются **только** нетронутые строки; замороженные пропускаются |
| **Эксперт без репо** | `/admin/imports/new` → вставить YAML → предпросмотр → подтвердить | ложится **черновиком** (`origin: ai`), та же защита заморозки |
| **Эксперт/редактор** | `/admin/lessons` → урок → редактировать (Lexxy) | **первая правка человека замораживает урок** (`origin: human`); каждая правка пишет **неизменяемую ревизию**; откат = новая ревизия |

Плюс **правки читателей:** «предложить правку» на уроке → редактор на
`/admin/lesson_suggestions` → одобряет → правка + ревизия.

Кто что может: `member` → `editor` (Эксперт: контент своих профессий) →
`administrator` (всё + публикация). Подробнее — `CLAUDE.md`.

## 8. Качество (QA) — две половины

- **(а) Механическая — правила, без суждения** (`lib/tasks/content.rake`):
  - `bin/rails content:audit` — теория, **написанная**, но без блока самопроверки
    `> [!ПРОВЕРЬ]`.
  - `bin/rails content:links` — ссылки-ресурсы, которые перестали открываться.
  - `bin/rails content:check` — обе разом.

  Намеренно **узкие**: полноту НЕ навязывают (не каждому уроку нужны практика/схема).

- **(б) Суждение — консольное ревью Claude Code** (`QA_REVIEW.md`): читает БД на
  чтение, судит ясность, техническую корректность, глубину, качество самопроверки,
  уместность схемы/практики. Итог — **предложения эксперту** (или отчёт). **ИИ
  предлагает — человек решает; в живой контент напрямую не пишем.**

## 9. Шпаргалка команд

```bash
bin/rails "content:import[svarshchik]"   # импорт одной профессии (черновик)
bin/rails content:import                 # импорт всего дерева сидов
bin/rails content:check                  # QA: audit + links разом
bin/rails content:audit                  # теория без самопроверки
bin/rails content:links                  # мёртвые ссылки (бьёт в сеть, медленно)
bin/rails test                           # все тесты
bin/rubocop                              # линт
```

## 10. Набор промптов (the kit)

- [`AUTHOR_PROFESSION.md`](AUTHOR_PROFESSION.md) — спроектировать roadmap и написать
  уроки целой профессии (формат, философия «библиотекаря», правила качества — **тут
  канон формата урока**).
- [`DEEPEN_LESSON.md`](DEEPEN_LESSON.md) — точечно углубить один готовый урок.
- [`LESSON_IMAGES.md`](LESSON_IMAGES.md) — инфографика к урокам (Nano Banana,
  фиксированный стиль).
- [`QA_REVIEW.md`](QA_REVIEW.md) — судящая половина QA (читаем БД → предложения).
- [`ARCHITECTURE_REVIEW.md`](ARCHITECTURE_REVIEW.md) — ревью кода/архитектуры самой
  фабрики (импортёры, заморозка, модель данных) на надёжность и удобство.

Код, который всё это исполняет: `lib/import_upsert.rb`, `lib/curriculum_importer.rb`,
`lib/curriculum_document.rb`, `app/models/concerns/importable.rb`,
`app/controllers/admin/imports_controller.rb`, `lib/tasks/content.rake`.
