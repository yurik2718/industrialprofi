## Ментальная модель: универсальный переводчик

Каждый ПЛК говорит на своём «языке» — Siemens S7 Comm, Mitsubishi MELSEC Protocol, Allen-Bradley EtherNet/IP. Чтобы SCADA понимала все ПЛК, раньше нужен был отдельный драйвер для каждого производителя. Это хаос: обновляешь SCADA — ломаются драйверы; меняешь ПЛК — пишешь новый коннектор.

**OPC UA** (OPC Unified Architecture) — это единый «язык», который понимают все. ПЛК экспортирует данные как OPC UA-сервер; SCADA, MES, ERP подключаются как OPC UA-клиенты. Все говорят на одном стандарте — независимо от производителя.

## Что такое OPC UA технически

OPC UA — это стандарт IEC 62541 (он же спецификация OPC Foundation), описывающий:

1. **Информационную модель** — как описывать данные и их структуру (объекты, переменные, методы, события)
2. **Сервисы** — операции, которые можно выполнять (Read, Write, Subscribe, Browse, Call)
3. **Транспорт** — как передавать данные (TCP бинарный, HTTPS, WebSockets, MQTT pub/sub)
4. **Безопасность** — аутентификация, авторизация, шифрование (встроены в стандарт)

### Информационная модель: адресное пространство

Данные в OPC UA организованы как **дерево узлов** (Address Space). Каждый узел имеет уникальный идентификатор — **NodeID**. Узлы бывают: объекты (Objects), переменные (Variables), методы (Methods).

```
Root
  └── Objects
        └── PLC_01 (Object)
              ├── Temperature_Reactor (Variable, NodeID=ns=1;s=PLC_01.Temp)
              │     ├── Value: 78.5 °C
              │     ├── DataType: Float
              │     └── AccessLevel: Read
              └── Pump_Run (Variable, NodeID=ns=1;s=PLC_01.Pump)
                    ├── Value: True
                    └── AccessLevel: ReadWrite
```

Клиент подключается и «обходит» (Browse) это дерево — находит нужные переменные, не зная заранее их расположения.

## Режимы доступа к данным

| Режим | Как работает | Когда использовать |
|-------|-------------|-------------------|
| **Read/Write** | Клиент читает/пишет значение по запросу | Редкое обращение, конфигурирование |
| **Subscription / MonitoredItems** | Сервер уведомляет клиента при изменении значения | Мониторинг в реальном времени: SCADA, историки |
| **Events** | Сервер отправляет события (аварии, изменения состояния) | Журналирование, SIEM |
| **Methods** | Клиент вызывает метод на сервере | Команды: «запустить насос», «сбросить аварию» |

**Subscription** — главный механизм OPC UA для промышленного мониторинга. Клиент говорит: «хочу получать Temperature_Reactor каждые 500 мс при изменении на 0.5°C». Сервер сам следит и отправляет обновления — не нужно постоянно «опрашивать» (polling).

## Безопасность: встроена, не добавлена

В отличие от Modbus, OPC UA имеет встроенную безопасность на уровне стандарта:

- **Аутентификация**: анонимная, логин/пароль, X.509-сертификаты
- **Авторизация**: права на чтение/запись отдельных узлов
- **Шифрование**: TLS 1.2+ (Basic256Sha256 и выше)
- **Подпись**: каждый пакет подписан, предотвращает подмену

> [!ВАЖНО]
> На производственных установках OPC UA **обязательно используй режим безопасности None только в изолированной тестовой среде**. На объекте — минимум SignAndEncrypt с Basic256Sha256. Сервер без аутентификации в режиме None — это открытая дверь в данные техпроцесса.

## Задание

Установи UaExpert (бесплатно на сайте Unified Automation). Подключись к публичному тестовому OPC UA-серверу OPC Foundation: `opc.tcp://opcuaserver.com:48010`.

1. Открой Address Space (Browse) — найди папку Objects. Что там?
2. Найди любую переменную с числовым значением. Запиши её NodeID, DataType и текущее значение.
3. Создай Subscription (Data Access View): перетащи найденную переменную на панель мониторинга. Как часто обновляется значение?

Запиши в журнал: NodeID переменной, её текущее значение, время обновления по Subscription.

Если нет доступа к интернету — опиши письменно: что такое NodeID и как OPC UA-клиент использует его для чтения конкретной переменной из сервера.
