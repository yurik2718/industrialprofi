Структурированный подход к диагностике сети — сверху вниз или снизу вверх по модели OSI — позволяет найти неисправность за 15 минут вместо 3 часов хаотичных проверок: в OT-сети каждая лишняя минута простоя стоит денег.
---
## Зачем нужна методология

«Пинг не идёт» — это симптом. Причин могут быть десятки: неверный VLAN, кабель без обжима, firewall с запрещающим правилом, неверная маска, петля в сети, отказ источника питания коммутатора. Без методологии инженер начинает проверять всё подряд — и тратит время.

**Методология OSI** (снизу вверх: L1 → L7) или (сверху вниз: L7 → L1) даёт последовательный список мест проверки. Найдена проблема на уровне — проверка остановлена, решение принято.

## Семь уровней — семь мест поиска

| Уровень | Вопрос | Инструмент |
|---------|--------|-----------|
| **L1 — Physical** | Кабель подключён? Горит ли link-LED? Нет ли повреждений? | Осмотр, кабельный тестер, OTDR |
| **L2 — Data Link** | Правильный ли VLAN? Видна ли MAC-адрес в таблице коммутатора? Нет ли петли (STP)? | `show mac address-table`, Wireshark (ARP), коммутатор CLI |
| **L3 — Network** | Правильный ли IP-адрес, маска, шлюз? Работает ли маршрутизация? | `ping`, `traceroute`, `show ip route` |
| **L4 — Transport** | Открыт ли TCP-порт? Нет ли блокировки firewall? | `telnet <ip> <port>`, Wireshark (TCP SYN/RST) |
| **L5-L6** | (Редко актуальны для OT-диагностики) | — |
| **L7 — Application** | Работает ли OT-протокол? Верен ли Unit ID? Правильный ли FC? | Modbus Poll, UaExpert, Proneta, Wireshark |

## Два подхода: снизу вверх vs сверху вниз

**Снизу вверх (Bottom-Up)**: начинай с L1 → проверяй каждый уровень. Применяй, когда нет никаких признаков на каком уровне проблема; когда это новая инсталляция.

**Сверху вниз (Top-Down)**: начинай с L7 (работает ли приложение?) → если нет, спускайся. Применяй, когда система работала и внезапно сломалась — скорее всего, проблема не на физическом уровне.

**Метод деления пополам (Divide and Conquer)**: начинай с L3 (`ping`). Если пинг идёт — проблема в L4-L7. Если нет — в L1-L2. Быстро сужает область поиска.

## Пошаговый алгоритм для OT

```
Симптом: ПЛК не отвечает / SCADA не получает данные
         │
1. Проверь физику (L1)
         │ link LED не горит?
         ▼ ДА → замени кабель, порт
         │
2. Проверь коммутатор (L2)
   - show mac address-table → MAC ПЛК есть? В каком VLAN?
   - Правильный ли access VLAN на порту ПЛК?
         │ MAC нет или VLAN неверный?
         ▼ ДА → исправь конфигурацию порта
         │
3. Проверь IP (L3)
   - ping IP ПЛК с инженерной станции
   - Правильная маска? Шлюз?
   - Есть ли маршрут между VLAN (на L3-коммутаторе)?
         │ ping не идёт?
         ▼ ДА → исправь IP/маску/маршрут/SVI на ядровом коммутаторе
         │
4. Проверь firewall / ACL (L4)
   - Открыт ли порт 502 (Modbus) / 4840 (OPC UA)?
   - telnet <IP ПЛК> 502 → соединение устанавливается?
         │ порт закрыт?
         ▼ ДА → добавь разрешающее правило firewall
         │
5. Проверь протокол (L7)
   - Modbus Poll / qModMaster → Unit ID совпадает с ПЛК?
   - Function Code поддерживается? Регистр существует?
   - Wireshark → запросы уходят? Ответы приходят? Exception codes?
         │ Exception Code?
         ▼ ДА → сверься со спецификацией устройства
         │
   РЕШЕНО
```

## Типичные неисправности в OT-сетях

| Симптом | Первое что проверить |
|---------|---------------------|
| ПЛК не пингуется, хотя кабель подключён | L2: VLAN на порту коммутатора |
| PROFINET устройство «не в топологии» | L2: DCP Identify в Wireshark — отвечает ли устройство? |
| Modbus работает, но данные неверные | L7: Unit ID, адрес регистра, масштабирование |
| Сеть периодически «падает» на секунды | L2: RSTP Topology Change (Wireshark: `stp`) — петля? |
| После добавления нового устройства PROFINET стал нестабильным | L2: дублирование IP-адреса (arpwatch, Wireshark ARP) |
| SCADA не может подключиться к ПЛК | L4: firewall правила, порт 102 (S7) или 502 (Modbus) |
| OPC UA сервер недостижим | L4: порт 4840; L7: сертификат, security policy |

> [!ПРИМЕР]
> **Реальный сценарий**: HMI показывает «Connection Lost» к ПЛК. Это было после ночного обслуживания.
>
> Инженер применил метод деления пополам:
> 1. `ping 10.20.1.11` (ПЛК) → не идёт. Значит L1-L3.
> 2. Link LED на коммутаторе горит → L1 ок.
> 3. `show mac address-table | include <MAC ПЛК>` → MAC в VLAN 20.
> 4. HMI находится в VLAN 30. SVI на ядровом коммутаторе: есть ли маршрут с VLAN 30 в VLAN 20?
> 5. `show running-config` — оказалось, SVI для VLAN 20 (`interface Vlan20`) был удалён во время обслуживания.
> 6. Добавил SVI обратно → ping пошёл → SCADA восстановилась.
>
> Весь процесс: 12 минут.

> [!ВАЖНО]
> В OT-сети всегда документируй каждое изменение во время диагностики. Если ты изменил VLAN на порту, правило firewall, или конфигурацию ПЛК — запиши что, когда, зачем. Без этого следующий инженер потратит часы на понимание «почему тут так?»

> [!ПРОВЕРЬ]
> 1. SCADA не получает данные от ПЛК, хотя раньше всё работало. С какого уровня OSI начнёшь диагностику и почему — снизу вверх или методом деления пополам?
> 2. `ping` к ПЛК идёт, но Modbus Poll не получает ответ. На каком уровне OSI проблема? Что конкретно проверишь?
> 3. После добавления нового коммутатора в кольцо MRP сеть стала периодически «падать» на 1–2 секунды. Что это может быть?

## Задание

Возьми любую неисправность из своей практики (или придумай гипотетическую по описанию выше).

Составь пошаговую карточку диагностики:
1. Опиши симптом.
2. Выбери стратегию (снизу вверх / сверху вниз / деление пополам) и обоснуй.
3. Пройди каждый уровень OSI — что именно проверяешь на каждом, каким инструментом.
4. На каком уровне нашёл (или предполагаешь) причину?
5. Что бы сделал для исправления?
