## Почему управляющие протоколы критичны в OT

В офисной среде коммутатор «работает и ладно» — проблемы видны сразу, пользователи жалуются. В OT-сети:
- Проблема с коммутатором = потеря связи с ПЛК = остановка производства
- Аварийные события происходят в 3 часа ночи; логи нужны для расследования
- ФСТЭК №239 требует аудита событий на всех значимых объектах (мера АУД.2)
- Форензика инцидента без синхронизированного времени бесполезна — чьё событие было первым?

Три протокола решают три задачи: **SNMP** — мониторинг состояния, **Syslog** — сбор событий, **NTP** — синхронизация времени.

## SNMP: Simple Network Management Protocol

**SNMP** — стандартный протокол управления сетевым оборудованием (RFC 1157, v3: RFC 3411-3418). Агент SNMP работает на каждом управляемом коммутаторе; менеджер (NMS) опрашивает агентов.

### SNMP v1/v2c vs v3

| Версия | Аутентификация | Шифрование | Применение в OT |
|--------|---------------|-----------|-----------------|
| v1 | Community string (plaintext) | Нет | Запрещено на значимых объектах КИИ |
| v2c | Community string (plaintext) | Нет | Только для несекретных систем вне КИИ |
| **v3** | MD5/SHA аутентификация | DES/AES шифрование | Требуется на объектах КИИ (АУД.2, ЗИС.20) |

> [!ВАЖНО]
> На объектах КИИ (и вообще в промышленных сетях, если ты следуешь лучшим практикам) — **только SNMPv3 с аутентификацией и шифрованием**. SNMP v1/v2c community «public» — открытая дверь: атакующий с доступом к сети читает всю конфигурацию оборудования.

### Что мониторить по SNMP в промышленной сети

| OID-группа | Что показывает |
|-----------|----------------|
| `interfaces` (IF-MIB) | Статус портов (up/down), счётчики ошибок, загрузка |
| `sysUpTime` | Время с последней перезагрузки — коммутатор перезагрузился? |
| `dot1dStp` (BRIDGE-MIB) | Состояние STP/RSTP портов, root bridge |
| `hrStorageTable` (HOST-RES-MIB) | Загрузка CPU, памяти |
| Vendor-specific MIB | Siemens SCALANCE: состояние MRP-кольца, питание, температура |

**Trap vs Poll**: 
- **Poll (GetRequest)** — NMS спрашивает устройство раз в N секунд (например, через Nagios, Zabbix, Icinga)
- **Trap** — устройство само посылает уведомление при событии (link down, температура выше порога)

В OT используй оба: Poll для регулярного мониторинга + Trap для немедленного уведомления о критических событиях.

## Syslog: сбор событий

**Syslog** (RFC 5424) — стандартный протокол пересылки log-сообщений с устройств на центральный сервер (UDP 514 или TCP 601).

Каждое Syslog-сообщение содержит:
- **Timestamp** — дата/время (критично: должно быть точным → нужен NTP)
- **Hostname** — откуда сообщение
- **Severity** (0–7): Emergency(0), Alert(1), Critical(2), Error(3), Warning(4), Notice(5), Info(6), Debug(7)
- **Facility** — категория источника (kernel, mail, local0-local7)
- **Message** — текст события

**Что важно логировать с сетевых устройств в OT:**

| Событие | Severity |
|---------|---------|
| Порт упал (link down) | Warning(4) |
| Порт поднялся (link up) | Notice(5) |
| RSTP topology change | Warning(4) |
| MRP ring open/closed | Warning(4) / Info(6) |
| Неудачная попытка авторизации | Error(3) |
| Изменение конфигурации | Notice(5) |
| Перегрузка CPU > 80% | Warning(4) |
| Температура выше порога | Warning(4) |

**Syslog-серверы для OT:**
- Graylog (open-source, бесплатно)
- Syslog-ng / rsyslog (Unix)
- MaxPatrol SIEM / KUMA (сертифицированные, для КИИ в России)

> [!ПРИМЕР]
> Настройка Syslog на SCALANCE XC216:
>
> В SINEMA или CLI:
> ```
> logging host 192.168.50.10  # IP Syslog-сервера
> logging trap warnings       # отправлять события Warning и выше
> logging facility local6     # метка источника
> ```
> После этого все link-down, ошибки конфигурации и auth-failures уходят на центральный сервер — можно искать по всем коммутаторам из одного места.

## NTP: синхронизация времени

**NTP (Network Time Protocol)** — протокол синхронизации времени по сети. Без синхронизации:
- Логи с разных устройств не сопоставить — у каждого своё «время»
- Расследование инцидента невозможно: «кто был первым?»
- PROFINET IRT и PTP (IEEE 1588) требуют точной синхронизации для RT-трафика
- Сертификаты X.509 (OPC UA TLS) проверяются по времени — неверное время = ошибка сертификата

### Иерархия NTP (stratum)

```
GPS / Атомные часы  ← Stratum 0 (эталон)
        │
  NTP Server  ← Stratum 1 (основной для предприятия)
        │
 NTP Server 2 ← Stratum 2 (резервный)
    │      │
 Коммутаторы, ПЛК, SCADA, HMI  ← Stratum 3
```

**Для промышленного объекта** — рекомендуется иметь внутренний NTP-сервер (или роутер как NTP-источник) с синхронизацией от внешнего источника. Прямое обращение каждого ПЛК к pool.ntp.org в OT-сети запрещено (нет выхода в интернет из OT-сегмента по правилам безопасности).

### NTP vs PTP (IEEE 1588)

| Протокол | Точность | Применение |
|----------|---------|-----------|
| **NTP** | 1–50 мс | Логи, аудит, управление — достаточно |
| **PTP (IEEE 1588)** | < 1 мкс | PROFINET IRT, TSN, точные измерения — требует поддержки коммутаторов |

Для большинства OT-сетей NTP достаточен для синхронизации логов. PTP нужен только там, где требуется субмиллисекундная точность (IRT, силовая электроника).

> 1. Почему SNMPv1/v2c опасны для OT-сети? Что конкретно может прочитать атакующий через SNMP v2c с community «public»?
> 2. Если два коммутатора зафиксировали событие с разницей времени в 5 минут — почему это проблема для расследования инцидента?

## Задание

Зайди на CLI или веб-интерфейс любого управляемого коммутатора (учебный, домашний роутер с SSH, или виртуальный коммутатор в Cisco Packet Tracer).

1. Найди настройки SNMP: какая версия включена? Есть ли community «public»? Если есть — это уязвимость, запиши это.
2. Найди настройки NTP или SNTP: синхронизирован ли коммутатор с NTP-сервером? Проверь текущее время.
3. Найди настройки Syslog: куда отправляются события? Если никуда — это пробел в безопасности.

Запиши в журнал:
- Какое оборудование использовал?
- Что нашёл по каждому из трёх протоколов?
- Какие конкретные проблемы безопасности или пробелы обнаружил?
- Что бы исправил в первую очередь?

Если нет реального оборудования — используй Cisco Packet Tracer: добавь коммутатор, сконфигурируй SNMPv3, NTP-клиент и Syslog-сервер (виртуальный), верифицируй работу командами `show snmp`, `show ntp status`, `show logging`.
