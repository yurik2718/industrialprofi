Пассивный мониторинг OT-сети — единственный способ обнаружить атаку, не нарушив работу производства: активные сканеры ломают ПЛК, поэтому OT-IDS всегда слушает трафик без отправки пакетов.
---
## Почему обычный IT-мониторинг не работает в OT

В IT-безопасности стандарт — активное сканирование (Nessus, OpenVAS), агенты на каждом хосте (EDR), сбор логов через agentful SIEM. В OT эти подходы создают проблемы:

- **Активное сканирование ломает ПЛК**: многие промышленные устройства не рассчитаны на интенсивный network scan — переполняются буферы, устройство перезагружается или зависает. Реальный случай: Nessus scan убил Modbus-слейв.
- **Агенты нельзя поставить на ПЛК**: у Siemens S7-1500, Beckhoff CX или реле Schneider нет ОС, на которой можно запустить агент.
- **Окна обслуживания редки**: ПЛК работает 24/7 годами. Поставить агент или перезагрузить устройство — это согласование, технологический останов.

Вывод: **основной метод мониторинга в OT — пассивное зеркалирование трафика (SPAN/TAP) + анализ**.

## Архитектура OT-мониторинга

```
  ПЛК ←────────────→ ET 200SP
   │
[SCALANCE XC216]
   │          │
  HMI       SPAN-порт
              │
         [OT-IDS сенсор]
              │
         [Collector / SIEM]
              │
         [SOC / Аналитик]
```

**SPAN-порт (Switched Port Analyzer)** или **TAP (Test Access Point)** — коммутатор копирует весь трафик с нужных портов на отдельный порт. IDS-сенсор подключен к этому порту и только слушает, не отправляет ничего.

**Пассивный OT-IDS** делает на основе трафика:
1. **Asset Discovery** — автоматически составляет список всех устройств (IP, MAC, вендор, прошивка)
2. **Protocol Baseline** — строит карту «нормального» трафика: кто с кем, каким протоколом, как часто
3. **Anomaly Detection** — отклонение от базовой линии = потенциальный инцидент
4. **Signature Detection** — известные атаки и вредоносные паттерны (IoC — Indicators of Compromise)

## Типовые аномалии в OT-трафике

| Аномалия | Что значит |
|----------|-----------|
| Новое устройство в сети | Несанкционированное подключение ноутбука или нового ПЛК |
| ПЛК посылает трафик не по расписанию | Возможная компрометация, попытка lateral movement |
| Modbus FC05/FC06/FC16 с неизвестного IP | Попытка записи в ПЛК с постороннего устройства |
| PROFINET DCP Identify Broadcast с нового узла | Разведка сети (нормально для ПНР, аномально в штатном режиме) |
| Резкий рост broadcast-трафика | Возможная petya/wanna-подобная активность или неправильная конфигурация |
| Попытка обращения к ПЛК из DMZ напрямую | Нарушение политики, возможный признак компрометации DMZ |
| Изменение конфигурации ПЛК (S7comm, Step 7 write) | Несогласованное изменение программы ПЛК |

## Реагирование на инциденты в OT (Incident Response)

Реагирование в OT отличается от IT: нельзя просто «изолировать хост» или «перезагрузить ПЛК» — это остановка производства.

**OT Incident Response приоритеты**:
1. **Безопасность персонала** — прежде всего выявить, нет ли угрозы физической безопасности
2. **Безопасность технологического процесса** — нет ли манипуляций с процессом
3. **Сохранение данных** — сбор логов и трафика до принятия мер
4. **Изоляция** — только если без неё невозможно остановить угрозу; технологический останов = большие потери

**Процесс при обнаружении аномалии**:
```
Аномалия обнаружена IDS
   ↓
Уведомление → ИБ-специалист + технолог
   ↓
Совместный анализ: реальная угроза или ложная тревога?
   ↓
Если угроза: эскалация → руководство + ИБ
   ↓
Изоляция сегмента (если критично) ИЛИ наблюдение с усиленным мониторингом
   ↓
Уведомление в НКЦКИ (если объект КИИ — обязательно, 24 часа)
   ↓
Расследование, устранение, анализ причин
```

## Сбор логов и SIEM для OT

**SIEM (Security Information and Event Management)** в OT собирает:
- Syslog от коммутаторов (авторизация, изменение конфигурации, link up/down)
- SNMP Trap от промышленного оборудования
- Логи с OT-IDS сенсора
- Windows Event Log от SCADA-серверов, historian, Jump Server

Для OT на объектах КИИ в России используются **сертифицированные SIEM**: MaxPatrol SIEM (Positive Technologies), KUMA (Kaspersky), RuSIEM. Они интегрируются с OT-IDS и имеют нужные сертификаты ФСТЭК.

## Задание

Настрой SPAN-порт на виртуальном коммутаторе и проанализируй OT-трафик в Wireshark.

**Если есть лабораторный стенд с ПЛК** или симулятором (ModRSsim2, SIMIT):
1. Настрой SPAN на коммутаторе (или port mirroring в VMware/GNS3).
2. Запусти захват в Wireshark на зеркальном порту.
3. Найди в трафике: Modbus запросы/ответы, ARP-запросы, PROFINET DCP (если есть) или широковещательный трафик.
4. Запиши в журнал: какие IP и MAC адреса видны, какие протоколы, есть ли что-то «необычное»?

**Если стенда нет** — используй учебный PCAP-файл (Wireshark Sample Captures на wiki.wireshark.org содержат Modbus и промышленные протоколы):
1. Скачай любой PCAP с Modbus или промышленным протоколом.
2. Открой в Wireshark.
3. Примени фильтр `modbus` или `profinet` — найди 3 типа разных пакетов.
4. Для каждого пакета запиши: IP источника/назначения, Function Code (для Modbus), что происходит по смыслу.
