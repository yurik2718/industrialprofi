# 📚 МЫ — БИБЛИОТЕКА КАЧЕСТВЕННЫХ МАТЕРИАЛОВ

### на книги, видео и официальную обязательную документацию

</div>

> **Наша позиция: профессионалом вас сделают практика и чтение скучной
> официальной документации — а не пересказы.** Поэтому мы не плодим ещё один
> курс «своими словами». Мы делаем главное: отбираем и ранжируем лучшие ссылки
> на **книги, видео и обязательные стандарты** (ПУЭ, ГОСТ, НАКС, ASME), а в
> самом уроке **кратко, на примерах и без воды объясняем основы** — ровно
> столько, чтобы первоисточник перестал пугать и стал понятен. Дальше работают
> руки и оригинал документа.

---

## Зачем

The Odin Project доказал: освоить full-stack веб-разработку можно бесплатно —
не через видеокурсы, а читая документацию и собирая реальные проекты.
**IndustrialProfi переносит эту модель в рабочие профессии.**

У сварщиков, электриков, специалистов КИПиА и других рабочих в СНГ
(Россия, Казахстан) нет структурированного, бесплатного пути развития на основе
стандартов. Официальные знания уже существуют — они в ГОСТ, ПУЭ, НАКС и ASME —
но разрознены, пугают объёмом и никем не разложены по полочкам.
IndustrialProfi собирает их в понятные карьерные роадмапы:

```
Профессия   →  Курс              →  Урок
(Электрик)     (Охрана труда)       (ПУЭ: Правила устройства электроустановок)
```

## Как устроен урок

Каждый урок следует одной честной структуре без воды — та же идея, что и в
задании Odin Project, но адаптированная под цех и площадку:

1. **ЗАЧЕМ** — одно-два предложения, почему это важно на работе.
2. **ОФИЦИАЛЬНЫЕ ДОКУМЕНТЫ** — выверенные ссылки на сами стандарты, с рангом
   (★ обязательно, ○ опционально). Без пересказа — ты читаешь первоисточник.
3. **ПРАКТИЧЕСКОЕ ЗАДАНИЕ** — конкретное, проверяемое.
4. **✓ Отметить выполненным** — бинарный прогресс, как в Odin. Сделано или нет.

Прогресс по курсу — это просто `выполнено / всего`. Никаких «в процессе», никакой
ложной геймификации — только ясная карта того, что ты знаешь и что дальше.

## Стек

Это намеренно «скучное» Rails-приложение **без шага сборки** — и в этом смысл.
Оно показывает, как много можно сделать на чистом Rails 8 и Hotwire, не имея
Node.js нигде.

- **Ruby 4.0 / Rails 8.1**
- **SQLite3** + Solid Queue, Solid Cache, Solid Cable
- **Hotwire** (Turbo + Stimulus) — серверный рендеринг HTML, без SPA
- **Чистый CSS**, отдаётся как есть через **Propshaft** — без Tailwind, PostCSS,
  бандлера и сборки. Каскад — это просто имена файлов в алфавитном порядке.
- **Importmap** — без Node, Webpack и Vite
- Авторизация через `has_secure_password` (bcrypt) — без Devise
- **Kamal 2** + Docker + Thruster для деплоя
- **Minitest** + фикстуры + Capybara
- Собственные шрифты Inter / Inter Tight через `@font-face`, цветовые токены
  OKLCH, единая тёмная тема — паттерны UI повторяют open-source приложения
  Basecamp (Writebook, Fizzy).

## Быстрый старт

Нужны Ruby 4.0.5 и Git. Никакого Node, Yarn или пайплайна ассетов настраивать
не нужно.

```bash
git clone https://github.com/yurik2718/industrialprofi.git
cd industrialprofi
bin/setup          # ставит гемы, готовит базу, заливает примеры данных
bin/dev            # запускает сервер на http://localhost:3000
```

Частые команды:

```bash
bin/rails test          # прогон тестов
bin/rails test:system   # системные тесты (Capybara)
bin/rails db:migrate    # миграции
bin/rubocop             # линтер
```

## Структура проекта

```
app/models/                 # доменные модели (Path, Course, Lesson, ...)
app/controllers/            # RESTful-контроллеры, рендерят ERB
app/views/                  # ERB-шаблоны + партиалы Turbo Frame/Stream
app/javascript/controllers/ # Stimulus-контроллеры
app/assets/stylesheets/     # весь CSS — по одному самодостаточному файлу на компонент
db/migrate/                 # миграции = источник истины для схемы
docs/                       # VISION.md, MVP.md, DEPLOY.md, промпты для контента
```

Иерархия контента:

```
Path (профессия)  →  Course  →  Lesson  →  Resource (ссылки на стандарты)
User  →  LessonCompletion  (бинарно: строка существует = урок пройден)
```

## План развития

IndustrialProfi выходит фазами — см. [docs/MVP.md](docs/MVP.md):

- **v0.1 — готово:** статический каталог (профессии → курсы → уроки, публично, SEO-first)
- **v0.2 — готово:** аккаунты, бинарный прогресс, личный кабинет, дневник
  практики, карта активности, правки от читателей + история версий, админка с ролями
- **v0.3 — дальше:** контент от сообщества (черновик → ревью → публикация),
  поиск, публичные профили

Публичная дорожная карта — на самом сайте, страница `/roadmap`.

Полная продуктовая мысль — в [docs/VISION.md](docs/VISION.md).

## Стать соавтором

> **Главное, что мы строим, — это не код, а сообщество, которое разложит знания
> рабочих профессий по полочкам.** Код — лишь оболочка. Сердце проекта — уроки,
> выверенные ссылки на стандарты и практические задания. Один сильный урок здесь
> ценится так же, как один сильный pull request.

Знания уже существуют — в ПУЭ, ГОСТ, НАКС, ASME. Но они разбросаны по тысячам
страниц PDF, пугают объёмом и никем не сведены в путь. **Ты не пишешь учебник с
нуля — ты прокладываешь маршрут по тому, что уже написано:** какой пункт читать,
в каком порядке, что сделать руками, чтобы проверить себя. Это под силу любому
практику.

Зачем вкладываться:

- **Твой урок прочитают тысячи** работающих людей — электриков, сварщиков,
  кипиашников, — которым сегодня негде взять структурированный бесплатный путь.
- **Контент остаётся свободным навсегда.** Уроки под лицензией CC BY-SA 4.0 —
  их нельзя «закрыть» и присвоить. Твой вклад работает на людей, а не на чей-то
  кошелёк.
- **Платформа спроектирована, чтобы пережить своего основателя.** Развитие — за
  экспертами: новую профессию приводит реальный практик, который становится её
  соавтором. Расширение идёт не «вширь по прихоти», а через живых мастеров.
- **Это путь к роли «Эксперт».** Системные, качественные правки — это заявка на
  то, чтобы вести целую профессию (см. лестницу ролей ниже).

### Лестница доверия

Роли — это не привилегии, а механизм делегирования, чтобы проект рос без узкого
горлышка в лице одного человека:

| Роль | Что может |
| --- | --- |
| **Участник** (`member`) | Учиться, вести дневник практики, **предлагать правки к любому уроку**. |
| **Эксперт** (`editor`) | Всё то же + редактировать уроки, курсы и профессии напрямую, принимать чужие правки, откатывать изменения. Это роль соавтора-практика, ведущего направление. |
| **Администратор** | Всё + управление пользователями и ролями. |

Путь наверх — через вклад: начни с правок, докажи качество, получи роль Эксперта
и веди свою профессию.

### Три способа внести вклад в контент

**1. Поправить урок прямо на сайте — без аккаунта.**
На странице любого урока есть кнопка «Предложить правку»: исправь опечатку,
битую ссылку, неточную формулировку или добавь недостающий стандарт. Правка
уходит на ревью Эксперту — регистрация не нужна. Самый низкий порог входа: если
заметил ошибку, исправь её за минуту. Каждая принятая правка попадает в
неизменяемую историю версий урока.

**2. Улучшить текст через репозиторий (pull request).**
Исходники всех уроков лежат в `db/seeds/curriculum/<профессия>/` обычным
Markdown. Открой нужный `.md`, поправь текст или фронтматтер, открой PR. Так
удобно для крупных правок — переписать раздел, добавить схему, обновить блок
ссылок.

```
db/seeds/curriculum/elektrik/
├── path.yml                                  # профессия: title, description, status
├── 01-elektrobezopasnost-i-ohrana-truda/
│   ├── course.yml                            # курс: title, slug, description, position, status
│   ├── 01-dopusk-i-gruppy/
│   │   ├── section.yml                       # раздел: просто title (заголовок внутри курса)
│   │   ├── 01-gruppy-dopuska.md              # урок
│   │   └── 02-srok-i-prodlenie.md
│   └── 02-...
└── 02-...
```

Структура урока — фронтматтер (заголовок, тип, ранжированные ссылки на
стандарты) + тело в Markdown:

```markdown
---
title: "ПУЭ: главный закон электрика — как устроен и как работать"
position: 1
kind: lesson                 # lesson | practice
resources:
  - title: "ПУЭ 7-е издание (полный текст)"
    url: "https://www.consultant.ru/document/cons_doc_LAW_98464/"
    kind: document            # document | article | video | tool
    required: true            # ★ обязательно / ○ опционально
    country_code: RU          # или пусто — если стандарт универсален
---
Одно честное предложение «зачем»: что ты получишь и почему это важно на работе.
(Это же — мета-описание урока для поиска, ≤155 символов.)
---
## Тело урока в Markdown
...
```

**3. Создать новую профессию.**
Это самый весомый вклад — и он начинается с разговора, а не с кода.
Открой **issue или Discussion** с предложением: какая профессия, на каких
стандартах строится, как выглядит маршрут от новичка до самостоятельного
специалиста. Так мы согласуем карту до того, как ты вложишь время. Дальше:

1. Создай папку `db/seeds/curriculum/<slug-профессии>/` и `path.yml`.
2. Разложи путь на курсы (`NN-курс/course.yml`) и разделы
   (`MM-раздел/section.yml`) — нумерованные префиксы задают порядок.
3. Напиши уроки (`.md`). Сидер идемпотентно подхватит дерево:
   `path.yml → course.yml → section.yml → урок`, а позиции уроков расставит по
   порядку обхода.
4. `bin/rails db:seed` локально, проверь в браузере, открой PR.

Профессии в работе можно публиковать постепенно: курс со `status: coming_soon`
показывается как «в разработке», пока наполняется.

### Планка качества урока

Это то, что отличает IndustrialProfi от очередного пересказа. Каждый урок:

- **Отвечает на «зачем».** Первая строка — честный ответ, что человек получит.
  Без «зачем» урок не читают.
- **Отправляет к первоисточнику, а не пересказывает его.** Ссылки на сами
  стандарты (ПУЭ, ГОСТ, НАКС, ASME), ранжированные: ★ обязательно, ○ опционально.
- **Называет конкретные инструменты.** Если у задачи есть де-факто стандартный
  софт (Modbus Poll, UaExpert, Wireshark…) — назови его прямо, а не «подберите
  подходящую программу».
- **Заканчивается проверяемым заданием.** Для практик — формат с целью, списком
  «понадобится», блоком безопасности, шагами, разделом «что сдать» и
  **самопроверкой** (да/нет-критерии со ссылкой на пункт стандарта). Это и есть
  «вопросы», на которые учащийся отвечает себе сам, доказывая, что освоил тему.

Готовые шаблоны и подробные промпты для авторов — в
[docs/CONTENT_PROMPT.md](docs/CONTENT_PROMPT.md) и
[docs/LESSON_DEEPEN_PROMPT.md](docs/LESSON_DEEPEN_PROMPT.md). Образцы практик в
формате-эталоне — например
`db/seeds/curriculum/elektrik/04-montazh/04-elektroshchity/03-sborka-shchita-praktika.md`.

### Код и формальности

Правишь код? Технические правила, поток PR и команды — в
[CONTRIBUTING.md](CONTRIBUTING.md). Коротко: делай как по умолчанию делает Rails,
смотри на open-source приложения Basecamp (Writebook, Fizzy), `bin/rails test` и
`bin/rubocop` должны быть зелёными.

Про модель: **платформа есть и остаётся open source** под AGPL-3.0. Небольшой
набор будущих хостинговых функций для работодателей (заверенные сертификаты,
биржа кандидатов/работодателей) может быть платным — это классический
**open-core**. При первом PR бот попросит принять лёгкий CLA (один клик): ты
сохраняешь авторские права на свой вклад, а проект получает свободу развиваться.
Детали — в [CONTRIBUTING.md](CONTRIBUTING.md).
